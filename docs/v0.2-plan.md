# TraceEval v0.2 Plan

## Objective

Ship the first version of TraceEval that is operationally useful to a real team already running evals.

The bar for `v0.2` is not better visuals. The bar is this:

- production incidents can be ingested automatically
- similar incidents can be clustered with enough quality to triage
- a reviewer can approve or reject generated eval cases
- approved cases can be exported into promptfoo
- a candidate prompt or retrieval version can be replayed against those cases
- the product can explain which regressions were introduced or fixed

## What v0.2 Must Do

### 1. Ingest real evidence

Supported source types for `v0.2`:

- tracing system events
- support conversation transcripts
- escalation tickets
- explicit negative feedback events

Required capabilities:

- connector health status
- idempotent ingest
- raw payload retention pointer
- normalized trace event creation
- PII redaction hooks before persistence

## 2. Cluster failures into reviewable buckets

The clusterer should combine:

- semantic similarity over transcript excerpts
- exact-match grouping for tool and error signatures
- user intent similarity
- policy-area heuristics
- time-window grouping for incident bursts

Each cluster needs:

- title
- confidence score
- likely root-cause label
- severity
- status
- owner
- related trace count

## 3. Generate eval proposals with lineage

Every generated case must include:

- source cluster
- source trace IDs
- minimal transcript slice or user prompt
- expected behavior
- assertion type
- reviewer status
- export status

The eval generator should prefer compact, reproducible cases over fully faithful transcript dumps.

## 4. Support human review

A reviewer should be able to:

- approve a generated case
- reject a generated case
- edit the description, expectation, or assertion type
- mark a case as duplicate or not reproducible
- attach notes explaining the decision

## 5. Replay candidate versions

The replay system should accept a baseline and a candidate definition, then produce:

- pass/fail or rubric deltas per case
- regressions introduced
- improvements captured
- likely attribution signals

Version metadata should support:

- prompt version
- model name
- retriever version
- tool manifest version
- policy pack version

## 6. Export into promptfoo

The export flow should support:

- selecting approved cases
- grouping by cluster or product area
- generating promptfoo config fragments
- emitting metadata for lineage and ownership
- optionally opening a PR in the eval repo later

## Non-Goals For v0.2

Do not overbuild these yet:

- fully automated remediation suggestions
- active learning prioritization
- multi-tenant enterprise RBAC
- warehouse-grade analytics
- autonomous PR merge workflows

Those can come after the core ingest -> cluster -> review -> replay -> export loop is proven.

## Proposed Architecture

### Backend services

- `api`: HTTP service for UI and integrations
- `ingest-worker`: normalizes source payloads into trace events
- `cluster-worker`: assigns traces into clusters
- `case-worker`: generates eval proposals from clusters
- `replay-worker`: runs baseline/candidate comparisons
- `export-worker`: writes promptfoo artifacts

### Storage

- Postgres for normalized entities and workflow state
- object storage for raw traces and transcript attachments
- vector index for semantic clustering and similarity lookup

### Suggested stack

- TypeScript
- Fastify or Express for API
- Postgres
- pgvector or equivalent vector extension
- Redis or Postgres-backed job queue
- promptfoo CLI integration for local export validation

## Data Model Additions

The current prototype tables are a good start, but `v0.2` should add these:

### `trace_artifacts`

Stores pointers to raw inputs and redacted payloads.

Fields:

- `id`
- `trace_event_id`
- `artifact_type`
- `storage_url`
- `redaction_status`
- `created_at`

### `cluster_labels`

Stores structured classifications for clusters.

Fields:

- `id`
- `failure_cluster_id`
- `label_type`
- `label_value`
- `confidence_score`
- `assigned_by`

### `case_reviews`

Captures human review lifecycle.

Fields:

- `id`
- `eval_case_id`
- `decision`
- `reviewer`
- `notes`
- `created_at`

### `release_versions`

Represents the artifact versions being compared.

Fields:

- `id`
- `environment`
- `prompt_version`
- `model_name`
- `retriever_version`
- `tool_manifest_version`
- `policy_pack_version`
- `created_at`

### `replay_case_results`

Stores replay results at per-case granularity.

Fields:

- `id`
- `replay_run_id`
- `eval_case_id`
- `baseline_score`
- `candidate_score`
- `delta`
- `verdict`
- `attribution_label`
- `details_json`

## UI Surfaces For v0.2

### 1. Ingestion view

Shows source health, lag, last successful ingest, and event counts.

### 2. Cluster queue

Shows cluster severity, confidence, likely cause, affected intents, and recent trace examples.

### 3. Case review screen

Shows proposed eval cases side by side with the source incident snippets and lets reviewers approve or edit.

### 4. Replay diff screen

Shows baseline vs candidate results, regressions, improvements, and likely cause attribution.

### 5. Export screen

Shows approved cases, promptfoo pack preview, and export history.

## API Plan

### Ingest

- `POST /api/ingest/source-events`
- `POST /api/ingest/support-transcripts`
- `POST /api/ingest/feedback-events`

### Clusters

- `GET /api/clusters`
- `GET /api/clusters/:id`
- `POST /api/clusters/:id/recompute`
- `POST /api/clusters/:id/assign-owner`

### Cases

- `GET /api/cases`
- `GET /api/cases/:id`
- `POST /api/clusters/:id/generate-cases`
- `POST /api/cases/:id/review`
- `PATCH /api/cases/:id`

### Replay

- `POST /api/replays`
- `GET /api/replays`
- `GET /api/replays/:id`
- `GET /api/replays/:id/results`

### Export

- `POST /api/exports/promptfoo`
- `GET /api/exports`
- `GET /api/exports/:id`
- `GET /api/exports/:id/content`

## Build Order

### Milestone 1: Real ingest plus normalization

Deliver:

- source adapters
- redaction pipeline
- normalized trace event persistence
- ingestion dashboard metrics

### Milestone 2: Clustering and triage

Deliver:

- clustering job
- cluster queue UI
- root-cause labels
- ownership assignment

### Milestone 3: Case generation and review

Deliver:

- eval proposal generator
- review workflow
- edit/reject/approve actions
- lineage metadata persistence

### Milestone 4: Replay and attribution

Deliver:

- baseline/candidate version registry
- replay execution engine
- per-case deltas
- attribution heuristics

### Milestone 5: promptfoo export

Deliver:

- export selection UI
- promptfoo config generation
- export audit log
- optional PR payload generation

## Success Metrics

TraceEval should track:

- `incident_to_case_hours`
- `approved_case_rate`
- `exported_case_rate`
- `escaped_incident_rate`
- `regression_detection_rate`
- `cluster_precision_reviewed`
- `time_to_root_cause_guess`

## Immediate Coding Priorities

If building `v0.2` now, the first tickets should be:

1. Replace SQLite schema with a Postgres-ready schema module.
2. Add a review state machine for eval cases.
3. Add a release/version registry for replay comparisons.
4. Add a replay results table and API.
5. Add promptfoo export batches with selectable case sets.
6. Add at least one real ingestion adapter.

## Why This Matters

If TraceEval only exports YAML, it is replaceable.

If TraceEval becomes the system that continuously converts production failures into trusted, replayable, reviewed eval assets, it becomes strategically important.
